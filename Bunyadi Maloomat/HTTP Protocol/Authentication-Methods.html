API Hacking ‚Üí Bunyadi Maloomat ‚Üí HTTP Protocol ‚Üí Authentication Methods

Authentication Methods woh mukhtalif tareeqay hain jin se ek API kisi user ya client application ki shanakht (identity) ki tasdeeq karti hai jo uske resources tak rasai haasil karne ki koshish kar raha hai.

Yeh bilkul waisa hai jaise kisi mahdood (restricted) ilaqay mein dakhil hone se pehle kisi ka ID card check karna.

Iska maqsad yeh yakeeni banana hai ke sirf jayaz aur ikhtiyar-yaafta (authorized) idaray hi API ke saath interact kar saken.

üìñ Authentication Kya Hai?

Definition:

Authentication woh process hai jis se verify kiya jata hai ke user ya application wahi hai jo wo hone ka daawa kar raha hai.


Yeh Authorization se mukhtalif hai:

  • ‚Üí Authentication: "Aap KAUN hain?" - Identity verification
  • ‚Üí Authorization: "Aapko KYA karne ki ijazat hai?" - Permission verification

üîë Common Authentication Methods

BASIC AUTH

1. Basic Authentication

Yeh Kaise Kaam Karta Hai:

Client har request ke saath username aur password (aksar base64 encoded) Authorization header mein bhejta hai.

Request Example
GET /api/users HTTP/1.1 Host: api.example.com Authorization: Basic dXNlcm5hbWU6cGFzc3dvcmQ= # Base64 encoded: username:password # Decoded: username:password
Istemal:
  • Saada, simple implementation
  • Internal APIs ke liye
  • Quick prototyping
  • Legacy systems
‚ö†Ô∏è Security Note:

Tanha itna secure nahin! Agar HTTPS par istemal na ho toh eavesdropping ka shikaar ho sakta hai. Base64 encoding encryption NAHI hai!

Pros & Cons:

‚úì Pros: Simple, widely supported, easy to implement

‚úó Cons: Insecure without HTTPS, credentials sent with every request

üîë
API KEYS

2. API Keys

Yeh Kaise Kaam Karta Hai:

Ek munfarid (unique) string ya token (API key) banaya jata hai aur client ko diya jata hai. Phir yeh key har API request ke saath bheji jaati hai.

Request Example
# Method 1: Header GET /api/data HTTP/1.1 Host: api.example.com X-API-Key: abc123xyz456def789ghi012 # Method 2: Query Parameter GET /api/data?api_key=abc123xyz456def789ghi012 HTTP/1.1 Host: api.example.com # Method 3: Request Body (rare) POST /api/data HTTP/1.1 Host: api.example.com Content-Type: application/json { "api_key": "abc123xyz456def789ghi012", "data": "..." }
Istemal:
  • Public APIs (Google Maps, Stripe)
  • Usage tracking aur rate limiting
  • Saada access control
  • Application-level authentication
‚ö†Ô∏è Security Note:

Agar ek API key compromise ho jaaye, toh key rakhne wala koi bhi shakhs API tak rasai haasil kar sakta hai! Application ko access deti hai, kisi khaas user ko nahin.

Pros & Cons:

‚úì Pros: Simple, good for tracking, easy rotation

‚úó Cons: Can be exposed in URLs/logs, no user-level auth

üé´
BEARER TOKEN / JWT

3. Bearer Tokens (OAuth 2.0 / JWT)

Yeh Kaise Kaam Karta Hai:

Jab ek user kamyabi se authenticate ho jata hai, server ek "bearer token" jari karta hai. Phir yeh token har agle request mein Authorization header mein Bearer <token> ke taur par shamil kiya jata hai.

JWT Token Example
# Login Request POST /api/auth/login HTTP/1.1 Content-Type: application/json { "username": "ali", "password": "secret123" } # Server Response HTTP/1.1 200 OK Content-Type: application/json { "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkFsaSIsImlhdCI6MTUxNjIzOTAyMn0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c" } # Subsequent Requests GET /api/profile HTTP/1.1 Host: api.example.com Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
JWT Structure:

JSON Web Token (JWT) teen parts par mushtamil hota hai:

  • Header: Algorithm aur token type
  • Payload: Claims (user data, expiry, etc.)
  • Signature: Cryptographically signed
Istemal:
  • Modern web applications
  • Mobile apps
  • Single Page Applications (SPAs)
  • Microservices authentication
‚ö†Ô∏è Security Note:

Bearer tokens kisi bhi shakhs ko rasai dete hain jiske paas woh hon! Inhe securely store karne aur HTTPS par transmit karne ki zaroorat hai. Token expiry set karein!

Pros & Cons:

‚úì Pros: Stateless, scalable, contains claims, widely adopted

‚úó Cons: Can't revoke easily, size larger than session ID

üç™
SESSION AUTH

4. Session-based Authentication

Yeh Kaise Kaam Karta Hai:

Jab ek user login karta hai, server ek munfarid session ID banata hai aur usay client ko wapas bhejta hai (aam taur par cookie ke taur par). Phir client har agle request ke saath yeh session ID bhejta hai.

Session Flow
# 1. Login POST /api/login HTTP/1.1 Content-Type: application/json { "username": "ali", "password": "secret123" } # 2. Server Response (Set Cookie) HTTP/1.1 200 OK Set-Cookie: sessionid=abc123xyz456; Path=/; HttpOnly; Secure # 3. Subsequent Requests (Cookie Auto-sent) GET /api/profile HTTP/1.1 Host: api.example.com Cookie: sessionid=abc123xyz456 # Server-side: # Session Store: { "abc123xyz456": { "user_id": 123, "username": "ali" } }
Istemal:
  • Traditional web applications
  • Multi-page applications
  • Server-rendered apps
  • Banking/financial applications
‚ö†Ô∏è Security Note:

Agar sahi tareeqay se implement na ho toh session hijacking, fixation, aur CSRF jaise hamlon ka shikaar ho sakta hai! HttpOnly aur Secure flags zaroori hain.

Pros & Cons:

‚úì Pros: Server control, easy revocation, small session ID

‚úó Cons: Requires server storage, not scalable for distributed systems

üîí
mTLS

5. Mutual TLS (mTLS)

Yeh Kaise Kaam Karta Hai:

Client aur server dono ek doosre ko digital certificates pesh karte hain taake apni shanakht ki tasdeeq kar saken. Yeh transport layer par ek mehfooz, authenticated connection qaim karta hai.

mTLS Flow
# Standard TLS/SSL: Client ‚Üí Server: "Hello" Server ‚Üí Client: Server Certificate Client verifies server # Mutual TLS (mTLS): Client ‚Üí Server: "Hello" Server ‚Üí Client: Server Certificate Client ‚Üí Server: Client Certificate Both verify each other! # Certificate Exchange: 1. Server requests client certificate 2. Client sends its certificate + private key signature 3. Server verifies client certificate against trusted CAs 4. Secure, authenticated connection established
Istemal:
  • High-security environments
  • Microservices architectures
  • Inter-service communication
  • Banking/government systems
  • IoT device authentication
‚ö†Ô∏è Security Note:

Bohat mazboot authentication faraham karta hai lekin isay set up aur manage karna ziyada pecheeda hota hai. Certificate management critical hai.

Pros & Cons:

‚úì Pros: Very secure, strong authentication, no credentials in requests

‚úó Cons: Complex setup, certificate management overhead

üîê
OAUTH 2.0

6. OAuth 2.0

Yeh Kaise Kaam Karta Hai:

OAuth 2.0 ek authorization framework hai jo third-party applications ko user ke resources tak limited access dene ki ijazat deti hai bagair password share kiye.

OAuth 2.0 Flow
# Authorization Code Flow: # 1. User Authorization Request https://auth.example.com/oauth/authorize? response_type=code& client_id=abc123& redirect_uri=https://app.com/callback& scope=read_profile # 2. User Approves ‚Üí Server Redirects https://app.com/callback?code=xyz789 # 3. Exchange Code for Token POST /oauth/token HTTP/1.1 Host: auth.example.com Content-Type: application/x-www-form-urlencoded grant_type=authorization_code& code=xyz789& client_id=abc123& client_secret=secret456& redirect_uri=https://app.com/callback # 4. Server Response { "access_token": "eyJhbGci...", "token_type": "Bearer", "expires_in": 3600, "refresh_token": "refresh_xyz", "scope": "read_profile" } # 5. Use Access Token GET /api/profile HTTP/1.1 Authorization: Bearer eyJhbGci...
OAuth 2.0 Roles:
  • Resource Owner: User (data ka malik)
  • Client: Application (access chahta hai)
  • Authorization Server: Tokens issue karta hai
  • Resource Server: Protected data store karta hai
Istemal:
  • Social login (Google, Facebook, GitHub)
  • Third-party integrations
  • Mobile app authentication
  • API access delegation
‚ö†Ô∏è Security Note:

OAuth 2.0 authentication protocol nahi hai, balkay authorization framework hai! OpenID Connect (OIDC) OAuth 2.0 par based authentication layer hai.

Pros & Cons:

‚úì Pros: Delegated access, no password sharing, flexible scopes

‚úó Cons: Complex implementation, multiple flows to understand

üî¥ Authentication Se Mutalliq Kamzoriyaan

üîë

1. Weak Credentials

Kamzor ya default credentials ka istemal:

  • Default passwords (admin/admin)
  • Simple passwords (123456, password)
  • Hardcoded credentials in code
  • No password complexity requirements
üö™

2. Broken Authentication

Authentication logic ki ghalat implementation:

  • No rate limiting on login attempts
  • Predictable session IDs
  • Session fixation vulnerabilities
  • Improper logout implementation
üîì

3. Token Vulnerabilities

Compromise shuda ya weak tokens:

  • JWT without signature verification
  • Exposed API keys in client code
  • Long-lived tokens without expiry
  • Tokens in URL parameters
‚öôÔ∏è

4. Misconfiguration

Ghalat security settings:

  • Missing HTTPS enforcement
  • Insecure cookie flags (no HttpOnly)
  • Weak JWT algorithms (none, HS256 with weak secret)
  • CORS misconfiguration
üé≠

5. Session Attacks

Session management ki kamzoriyaan:

  • Session hijacking via XSS
  • Session fixation attacks
  • No session timeout
  • Session data in URLs
üîç

6. Information Disclosure

Authentication process mein maloomat leak:

  • Different responses for valid/invalid users
  • Verbose error messages
  • Timing attacks possible
  • User enumeration via responses

‚úÖ Best Practices

Security Best Practices

üîí
1. Always Use HTTPS

Credentials aur tokens ko hamesha encrypted connection par bhejein. HTTP par kabhi sensitive data na bhejein.

üí™
2. Strong Password Policy

Password complexity requirements enforce karein: minimum length, uppercase, lowercase, numbers, symbols.

‚è∞
3. Token Expiration

Har token ke liye expiry time set karein. Access tokens short-lived (15-60 min), refresh tokens long-lived.

üîÑ
4. Token Rotation

API keys aur secrets ko regularly rotate karein. Compromised keys ko immediately revoke karein.

üö¶
5. Rate Limiting

Login endpoints par rate limiting implement karein taake brute force attacks prevent hon.

üç™
6. Secure Cookies

Session cookies mein hamesha HttpOnly, Secure, aur SameSite flags set karein.

üîê
7. Multi-Factor Auth (MFA)

Sensitive operations ke liye 2FA/MFA implement karein for extra security layer.

üìä
8. Logging & Monitoring

Authentication attempts, failures, aur suspicious activities ko log aur monitor karein.

üîç
9. Input Validation

User credentials ko properly validate aur sanitize karein SQL injection prevent karne ke liye.

üéØ API Hacking Se Talluq

Authentication Methods ko samajhna API hacking ke liye bohat zaroori hai kyunke kamzoriyan aksar in wajahat se paida hoti hain:

üîì

Authentication Bypass

Authentication logic ko bypass kar ke unauthorized access:

  • SQL injection in login forms
  • JWT signature bypass
  • Session fixation
  • Weak password reset mechanism
üé≠

Token Manipulation

Tokens ko modify kar ke privileges escalate karna:

  • JWT payload tampering
  • Algorithm confusion attacks
  • Token replay attacks
  • Forged signatures
üîë

Credential Stuffing

Leaked credentials ka istemal:

  • Automated login attempts
  • Using leaked password databases
  • No account lockout mechanism
  • Weak CAPTCHA implementation

üéì Learning Path: Roadmap mein "Authentication Bypass" aur "Session Token Manipulation" jaise topics shamil hain, jo in authentication mechanisms ki kamzoriyon ka faida uthane se mutalliq hain. Pehle authentication methods ko samajhna, phir unke vulnerabilities exploit karna seekhna!

üìù Mukhtasar

Key Takeaways:

  • ‚Üí Authentication: Identity verification - "Aap kaun hain?"
  • ‚Üí 6 Main Methods: Basic Auth, API Keys, Bearer Tokens, Sessions, mTLS, OAuth 2.0
  • ‚Üí Security Levels: Basic Auth (Low) ‚Üí mTLS (Very High)
  • ‚Üí Common Vulnerabilities: Weak credentials, broken auth, token issues
  • ‚Üí Best Practices: HTTPS, token expiry, rate limiting, secure cookies
  • ‚Üí JWT: Most popular modern approach - stateless, scalable
  • ‚Üí OAuth 2.0: Authorization framework, not authentication protocol
  • ‚Üí API Hacking: Authentication bypass, token manipulation, credential stuffing

üéØ Conclusion: Authentication methods ko samajhna API security ki bunyad hai. Har method ki strengths aur weaknesses ko jaan kar, aap behtar tareeqe se secure APIs bana sakte hain aur vulnerabilities ko identify kar sakte hain. API hacking mein in methods ki kamzoriyon ko exploit karna aik ahem skill hai!
