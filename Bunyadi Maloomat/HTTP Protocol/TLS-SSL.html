TLS aur SSL (Transport Layer Security aur Secure Sockets Layer) aise cryptographic protocols hain jo computer network par secure communication faraham karne ke liye banaye gaye hain.

API hacking ke silsile mein, inhein samajhna bohot zaroori hai kyunki APIs aksar in protocols par depend karti hain taake data ko safar ke dauran mehfooz rakha jaa sake.

üìñ SSL aur TLS Kya Hain?

SSL vs TLS Comparison

üîì

SSL (Secure Sockets Layer)

Kya hai: Netscape ne banaya hua asli protocol tha. Web communication ko secure karne ke liye bohot use hota tha.


Status: DEPRECATED ‚ùå

  • SSL 1.0 - Kabhi release nahi hua
  • SSL 2.0 - 1995 (Retired)
  • SSL 3.0 - 1996 (Retired)
  • Security flaws ki wajah se abandoned
  • Vulnerable to multiple attacks
üîí

TLS (Transport Layer Security)

Kya hai: SSL ka naya aur behtar version hai. SSL ke bohot se kamzorion ko door karta hai aur zyada secure hai.


Status: CURRENT STANDARD ‚úì

  • TLS 1.0 - 1999 (Deprecated)
  • TLS 1.1 - 2006 (Deprecated)
  • TLS 1.2 - 2008 (Secure)
  • TLS 1.3 - 2018 (Most Secure)
  • Modern systems ka standard

‚ö†Ô∏è Important Note: Jab log aaj "SSL" ka zikr karte hain, to woh aksar darasal TLS ki baat kar rahe hote hain, kyunki zyada tar modern systems TLS hi istemal karte hain. Term "SSL" ab bhi commonly use hota hai, lekin actual implementation TLS hai.

Protocol Version Release Year Status Security Level
SSL 2.0 1995 Deprecated Insecure - Multiple vulnerabilities
SSL 3.0 1996 Deprecated Insecure - POODLE attack
TLS 1.0 1999 Deprecated Weak - Not recommended
TLS 1.1 2006 Deprecated Weak - Not recommended
TLS 1.2 2008 Active Secure - Widely supported
TLS 1.3 2018 Recommended Most Secure - Latest standard

ü§ù TLS/SSL Handshake Process

1

Client Hello

Client "hello" kehta hai aur batata hai ke woh kaun se TLS versions aur cipher suites support karta hai. Random data aur session ID bhi bhejta hai.

Client ‚Üí Server: "Hello! Main TLS 1.2, TLS 1.3 support karta hun Cipher Suites: AES-256-GCM, ChaCha20-Poly1305"
2

Server Hello

Server apne pasandeeda TLS version, cipher suite ke saath jawab deta hai aur apna digital certificate bhejta hai. Server bhi random data generate karta hai.

Server ‚Üí Client: "Hello! Main TLS 1.3 use karunga Cipher Suite: AES-256-GCM Yeh mera Certificate hai (Public Key included)"
3

Certificate Verification

Client server ke certificate ko verify karta hai. Certificate mein server ki public key hoti hai aur yeh ek bharosemand Certificate Authority (CA) se sign kiya hua hota hai.

Client checks: ‚úì Certificate valid hai? ‚úì CA trusted hai? ‚úì Domain name match karta hai? ‚úì Certificate expire nahi hua?
4

Key Exchange

Agar certificate valid hota hai, to client aur server session ke liye ek shared secret key par agree karte hain. Yeh key exchange secure tareeqe se hota hai using asymmetric encryption.

Client + Server ‚Üí Generate: Pre-master secret ‚Üí Master secret ‚Üí Session Keys (Using Diffie-Hellman or RSA)
5

Secure Communication Begins

Handshake mukammal! Ab client aur server symmetric encryption ka istemal karke communicate karte hain using shared session key. Yeh kahin zyada tez hai!

All future messages: Encrypted with Shared Session Key ‚úì Fast symmetric encryption ‚úì Data confidentiality ‚úì Data integrity

üîë Key Concepts

üîê

Symmetric Encryption

Handshake ke baad, client aur server ek hi secret key ka istemal karte hain encryption aur decryption ke liye. Yeh bohot tez hai aur continuous data transfer ke liye ideal.

üîë

Asymmetric Encryption

Handshake mein public/private key pair ka istemal hota hai. Public key se encrypt kiya jata hai, private key se decrypt. Initial key exchange ke liye secure.

‚úÖ

Data Integrity

TLS/SSL data integrity ko yaqeeni banata hai using HMAC (Hash-based Message Authentication Code). Safar ke dauran data mein koi tabdeeli detect ho jati hai.

üìú

Digital Certificates

Server ki identity verify karne ke liye. Certificate Authority (CA) se sign kiya hua, jisme server ki public key aur identity maloomat hoti hai.

üé≤

Cipher Suites

Algorithms ka set jo encryption, key exchange, aur authentication methods define karta hai. Maslan: TLS_AES_256_GCM_SHA384

‚è±Ô∏è

Session Keys

Har session ke liye unique temporary keys generate hote hain. Perfect Forward Secrecy (PFS) ensure karta hai ke past sessions compromise nahi hote.

3 Main Security Benefits

üîí

1. Confidentiality (Raazdaari)

API client aur API server ke darmiyan exchange hone wala sensitive data niji rahe:


  • ‚Üí Authentication tokens encrypted rahte hain
  • ‚Üí Niji maloomat (personal data) safe hai
  • ‚Üí Maali data (financial data) protected
  • ‚Üí API keys aur secrets hidden
  • ‚Üí Ghair-ikhtiyar shakhsiyat sun nahi sakti
‚úÖ

2. Integrity (Salamat Rehna)

API ko bheja gaya aur wahan se mila hua data transmission ke dauran tabdeel nahi hua:


  • ‚Üí Requests manipulate nahi ho sakti
  • ‚Üí Responses tamper nahi ho sakti
  • ‚Üí Data corruption detect ho jati hai
  • ‚Üí MITM attacks prevent hote hain
  • ‚Üí Hamlawaron ko data badalne se rokta hai
üé´

3. Authentication (Tasdeeq)

Server ka certificate client ke liye server ki tasdeeq karta hai:


  • ‚Üí Client yaqeen kar sakta hai server legitimate hai
  • ‚Üí Phishing attacks se bachao
  • ‚Üí Fake servers identify ho jate hain
  • ‚Üí Domain validation automatic
  • ‚Üí Naqli hamlawar se bachata hai

üî¥ Security Vulnerabilities aur Attack Vectors

‚öôÔ∏è

1. Misconfigurations

TLS/SSL ghalat tarike se configure hone par vulnerabilities paida hoti hain:

  • Purane SSL versions: SSL 2.0, 3.0, TLS 1.0 use karna
  • Kamzor cipher suites: Weak algorithms allow karna
  • Expired certificates: Certificate validity check na karna
  • Self-signed certificates: Production mein use karna
  • Missing HSTS: HTTP Strict Transport Security na hona
Vulnerable Configuration: SSLProtocol +SSLv3 +TLSv1 ‚ùå CipherSuite: RC4-SHA ‚ùå Secure Configuration: SSLProtocol -all +TLSv1.2 +TLSv1.3 ‚úì CipherSuite: ECDHE-RSA-AES256-GCM-SHA384 ‚úì
üîç

2. Proxying Traffic (MITM)

Burp Suite jaise tools ek Man-in-the-Middle proxy ke taur par kaam karte hain:

  • Certificate Install: Burp ka CA certificate client par install karna
  • Traffic Intercept: HTTPS traffic ko decrypt aur inspect karna
  • Request Modify: Encrypted data ko modify karna
  • Response Analyze: Server responses ka tajziya karna
  • Yeh batata hai ke kaise TLS bypass kiya ja sakta hai testing ke liye
Normal Flow: Client ‚ü∑ [TLS Encrypted] ‚ü∑ Server With Burp Suite: Client ‚ü∑ [TLS] ‚ü∑ Burp ‚ü∑ [TLS] ‚ü∑ Server ‚Üì Burp sees everything ‚Üì Plaintext Data Visible!
üìå

3. Certificate Pinning Bypass

Kuch applications certificate pinning implement karti hain MITM se bachne ke liye:

  • Kya hai: App sirf specific certificates ko trust karti hai
  • Purpose: Trusted CAs se bhi MITM prevent karna
  • Bypass: Advanced technique using tools like Frida, Objection
  • Root detection: Device rooted/jailbroken check karna
  • Security testing ke liye zaruri skill
Certificate Pinning: App only trusts: SHA256(api.example.com cert) Bypass Techniques: 1. Frida script injection 2. SSL unpinning tools 3. Patching the app 4. Runtime hooking
üíÄ

4. Known TLS/SSL Attacks

Famous attacks jo TLS/SSL vulnerabilities exploit karte hain:

  • POODLE: SSL 3.0 vulnerability (2014)
  • BEAST: TLS 1.0 CBC cipher attack (2011)
  • Heartbleed: OpenSSL memory leak (2014)
  • DROWN: SSLv2 protocol attack (2016)
  • FREAK: Export cipher vulnerability (2015)
Famous Vulnerabilities: CVE-2014-0160 (Heartbleed) - Memory leak CVE-2014-3566 (POODLE) - SSL 3.0 downgrade CVE-2016-0800 (DROWN) - SSLv2 attack CVE-2015-0204 (FREAK) - Weak export ciphers
‚¨áÔ∏è

5. Downgrade Attacks

Attacker client aur server ko kamzor protocol use karne par majboor karta hai:

  • Protocol Downgrade: TLS 1.3 se TLS 1.0 tak
  • Cipher Downgrade: Strong se weak cipher tak
  • MITM Required: Handshake manipulation
  • Prevention: TLS_FALLBACK_SCSV mechanism
  • Purane protocols disable karke rokna
Attack Scenario: 1. Client offers: TLS 1.3, 1.2, 1.1, 1.0 2. Attacker intercepts and removes: TLS 1.3, 1.2 3. Server sees only: TLS 1.1, 1.0 4. Connection downgrades to TLS 1.0 5. Attacker exploits TLS 1.0 vulnerabilities
üîß

6. Implementation Flaws

Server-side ya client-side implementation ki ghaltiyaan:

  • Certificate Validation: Theek se verify na karna
  • Hostname Verification: Domain name match na karna
  • Mixed Content: HTTP aur HTTPS dono ka istemal
  • Insecure Renegotiation: Handshake mein kamzori
  • Forward Secrecy: PFS enable na hona
Common Implementation Mistakes: ‚ùå curl_setopt(CURLOPT_SSL_VERIFYPEER, false); ‚ùå ctx.set_verify_mode(ssl::verify_none); ‚ùå on HTTPS page ‚úì Always verify certificates ‚úì Enable hostname verification ‚úì Use HTTPS everywhere ‚úì Enable Forward Secrecy

‚úÖ Best Practices

Security Best Practices:

  • ‚úì Sirf TLS 1.2 aur 1.3 enable karein - Purane versions disable
  • ‚úì Strong cipher suites use karein - AES-256-GCM, ChaCha20-Poly1305
  • ‚úì Valid certificates use karein - Trusted CA se sign kiya hua
  • ‚úì HSTS enable karein - HTTP Strict Transport Security

  • ‚úì Monitor aur audit karein - Regular security assessments
