Yeh API Hacking ke liye ahem hai kyunki GraphQL naye hamlon ke rastay (attack vectors) paida karta hai aur REST APIs ke muqable mein mukhtalif reconnaissance aur exploitation techniques ki zaroorat hoti hai.

Graph Query Language

üéØ GraphQL Kya Hai?

üìñ Tareef:

GraphQL aapki API ke liye ek query language hai aur un queries ko aap ke mojooda data se poora karne ke liye ek runtime hai.


Yeh clients ko bilkul wohi data talab karne ki ijazat deta hai jo unhein chahiye, na zyada na kam, sirf aik endpoint par aik single query bhej kar.

REST VS GraphQL

üåê REST APIs

  • Multiple Endpoints: /users, /products, /orders
  • Fixed Data Structure: Server decide karta hai kya data bhejein
  • Over-fetching: Zaroorat se zyada data mil sakta hai
  • Under-fetching: Multiple requests ki zaroorat
  • GET, POST, PUT, DELETE: Different HTTP methods

‚ö° GraphQL APIs

  • Single Endpoint: /graphql
  • Flexible Data Structure: Client decide karta hai kya chahiye
  • No Over-fetching: Sirf required data milta hai
  • No Under-fetching: Ek query mein sab mil jata hai
  • Queries & Mutations: POST method se operations

‚ú® A Query Language for Your API

Single Endpoint: REST ke bar-aks, jahan aapke paas aam tor par kayi endpoints hote hain (maslan, /users, /products, /orders), GraphQL sirf ek single endpoint istemal karta hai (maslan, /graphql).

üîç Kaise Kaam Karta Hai:

Clients is single endpoint par ek "query" bhejte hain, jisme woh bilkul wahi data aur usi structure mein specify karte hain jiski unhein zaroorat hoti hai.

GraphQL Query Example
{ user(id: "123") { name email posts { title comments { text } } } }

üéØ Result: Is ek query mein aapko user ki tafseelat, uski posts, aur har post ke comments mil jayenge - sab kuch ek hi request mein!

üéØ Bilkul Wahi Maangein Jiski Aapko Zaroorat Hai

üö´

No Over-fetching

Zaroorat Se Zyada Data Hasil Na Karna:

REST mein, agar aap users ki list mangte hain, toh API shayad user ki saari tafseelat bhej de, bhale hi aapko sirf naam chahiye.

GraphQL: Aap users { name } specify kar sakte hain, aur aapko sirf naam milenge.

‚úÖ

No Under-fetching

Zaroorat Se Kam Data Hasil Na Karna:

REST mein, agar aapko ek user aur uski posts chahiye, toh aapko do alag requests karni paden.

GraphQL: Sab kuch ek query mein: user(id: "123") { name posts { title } }

üìä Practical Example

REST Approach:

Multiple Requests Required
// Request 1: Get user GET /users/123 // Request 2: Get user's posts GET /users/123/posts // Request 3: Get comments for each post GET /posts/1/comments GET /posts/2/comments ...

GraphQL Approach:

Single Request - Everything
// One Query - All Data query { user(id: "123") { name email posts { title comments { text author } } } }

üîß Strongly Typed Schema

üìú Schema - The Contract

GraphQL APIs ko ek "schema" se define kiya jata hai jo tamaam data types aur operations ko bayan karta hai.

Yeh schema client aur server ke darmiyan ek contract ka kaam karta hai.

üéØ Schema Ki Khasusiyat:

  • Data Types: Har field ki type clearly defined hoti hai (String, Int, Boolean, etc.)
  • Available Operations: Queries, mutations, aur subscriptions
  • Validation: Automatic validation - galat query reject ho jati hai
  • Auto-completion: Developer tools mein intelligent suggestions
  • Documentation: Schema khud documentation provide karta hai
Schema Example
type User { id: ID! name: String! email: String! posts: [Post] } type Post { id: ID! title: String! content: String author: User! comments: [Comment] } type Query { user(id: ID!): User users: [User] }

‚öôÔ∏è Operations (Amal)

üîç

Queries

Data hasil karne ke liye istemal hote hain.

REST mein GET method ki tarah.

query { users { name email } }
‚úèÔ∏è

Mutations

Data ko tabdeel karne ke liye istemal hote hain (banana, update, delete).

REST mein POST, PUT, DELETE ki tarah.

mutation { createUser( name: "Ali" email: "ali@test.com" ) { id name } }
üîî

Subscriptions

Real-time data updates ke liye istemal hote hain.

Server client ko data bhejta hai jab tabdeeliyan hoti hain.

subscription { newPost { title author { name } } }

üîí API Hacking Mein Yeh Kyun Ahem Hai?

‚ö†Ô∏è GraphQL Attack Vectors

Security ke lehaaz se, GraphQL naye hamlon ke rastay (attack vectors) paida karta hai:

üîç 1. Introspection Queries

GraphQL mein clients schema ko query kar sakte hain taake woh samajh saken ke kya data available hai.

Risk: Agar introspection enabled hai, toh hamlawaar puri API structure ki maloomat hasil kar sakta hai - saare types, fields, aur operations.

üì¶ 2. Batching Attacks

GraphQL multiple queries ko ek hi request mein bhejne ki ijazat deta hai.

Risk: Hamlawar is feature ka abuse kar ke bohat saari queries ek saath bhej sakta hai, jo server par load barha sakta hai ya rate limiting ko bypass kar sakta hai.

üîÅ 3. Alias Abuse

GraphQL aliases ki ijazat deta hai - same field ko alag naam se multiple baar query karna.

Risk: Hamlawar same field ko hazaron baar query kar sakta hai, jo DoS attack ban sakta hai.

üåÄ 4. Resource Exhaustion (Deeply Nested Queries)

GraphQL deeply nested queries ko support karta hai.

Risk: Bohot zyada nested queries server ko overwhelm kar sakti hain aur DoS attack ban sakti hain.

query { user { posts { comments { author { posts { comments { # Infinite nesting... } } } } } } }

üö™ 5. Authorization Issues

Field-level access control GraphQL mein complex hai.

Risk: Agar har field par proper authorization checks nahi hain, toh sensitive data leak ho sakta hai.

üìÑ 6. Information Disclosure

Error messages aur introspection se zyada information leak ho sakti hai.

Risk: Database structure, backend implementation details, aur internal field names expose ho sakte hain.

üìö Important: Is roadmap mein "WebSockets Aur GraphQL" ke andar "GraphQL Attacks" ka topic in khaas kamzoriyon ko detail mein cover karta hai.

GraphQL kaise kaam karta hai, yeh samajhna in kamzoriyon ko pehchanne aur unka faida uthane ka pehla qadam hai.

‚ú® GraphQL Ki Taqat Aur Kamzoriyaan

üëç Fawaid:

  • Single endpoint - API management aasaan
  • Client ko complete control - exactly wahi data jo chahiye
  • No over-fetching - bandwidth save hoti hai
  • No under-fetching - kam requests, better performance
  • Strongly typed schema - better validation aur tooling
  • Real-time capabilities - subscriptions ke through
  • Self-documenting - schema se documentation automatic

‚ö†Ô∏è Security Challenges:

  • ‚úó Introspection se puri API structure expose ho sakti hai
  • ‚úó Complex queries se DoS attacks possible
  • ‚úó Field-level authorization implement karna mushkil
  • ‚úó Rate limiting aur query cost calculation complex
  • ‚úó Traditional WAFs GraphQL ko properly protect nahi kar pate

üìù Mukhtasar

Key Takeaways:

  • GraphQL ek query language hai APIs ke liye - Facebook ne develop kiya
  • Single Endpoint: /graphql - sab operations yahan hote hain
  • Flexible Queries: Client decide karta hai kya data chahiye
  • No Over/Under-fetching: Exactly required data milta hai
  • Strongly Typed Schema: Clear contract between client and server
  • Three Operations: Queries (read), Mutations (write), Subscriptions (real-time)
  • Security Risks: Introspection, Batching, Nested Queries, Authorization
  • Modern Alternative: REST ka powerful replacement, lekin naye security challenges

üéØ Conclusion: GraphQL ko samajhna API hacking mein zaroori hai kyunki yeh REST se mukhtalif attack vectors provide karta hai. GraphQL ki structure, schema, aur features ko samajh kar hi aap iske vulnerabilities ko effectively exploit kar sakte hain.
