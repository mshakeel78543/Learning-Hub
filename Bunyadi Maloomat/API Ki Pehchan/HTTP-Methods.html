HTTP Verbs - API Communication Ka Asaas

API Hacking ‚Üí Bunyadi Maloomat ‚Üí API Ki Pehchan ‚Üí HTTP Methods

HTTP methods, jinhain HTTP verbs bhi kehte hain, is baat ka bunyadi hissa hain ke clients (jaise aapka web browser ya mobile app) servers (jahan APIs mojood hoti hain) ke saath kaise communicate karte hain.

Ye us action ki nishandahi karte hain jo ek URL se pehchanay gaye resource par perform karna hai. Inhain aap ek API ko bheji ja rahi request ki "qism" samajh sakte hain.

üìñ HTTP Methods Kya Hain?

Definition:

HTTP methods verbs hain jo yeh define karte hain ke ek resource par kaunsa action perform karna hai. Yeh client aur server ke darmiyan communication ke liye standardized commands hain.


Formula: HTTP Method + Endpoint = Complete Request

GET https://api.example.com/users

‚Üë Method + URL = Action

üìö Detailed HTTP Methods

GET
‚úì Safe ‚úì Idempotent

üéØ Maqsad (Purpose):

Kisi khaas resource se data retrieve karna.

‚≠ê Khaasiyat (Characteristics):

  • Safe: Server ki state ko change nahi karte
  • Idempotent: Kai baar same request bhejne se wahi nateeja
  • Sirf data retrieve karna chahiye, koi doosra asar nahi
  • Parameters URL mein (query strings) bheje jate hain
  • Cacheable hai - performance ke liye

üí° Misaal (Examples):

GET /products ‚Üí Products ki list lena GET /users/123 ‚Üí User 123 ka profile fetch karna GET /orders?status=pending ‚Üí Pending orders hasil karna

üîí Hacking Mein:

Attackers GET requests ka istemal karte hain:

  • Resources ko enumerate karne ke liye
  • Information jama karne ke liye
  • IDOR testing - URL parameters manipulate karke
  • SQL Injection - Query parameters mein
  • Directory traversal - Path manipulation
POST
‚úó Not Safe ‚úó Not Idempotent

üéØ Maqsad (Purpose):

Kisi khaas resource par data submit karna taake usay process kiya ja sake. Aksar ek naya resource banta hai.

‚≠ê Khaasiyat (Characteristics):

  • Not Safe: Server ki state change hoti hai
  • Not Idempotent: Har request naya resource bana sakti hai
  • Data request body mein bheja jata hai
  • Baar baar bhejne se multiple resources ban sakte hain
  • Generally not cacheable

üí° Misaal (Examples):

POST /users Body: { "name": "Ali", "email": "ali@test.com" } ‚Üí Naya user account banana POST /orders Body: { "product_id": 123, "quantity": 2 } ‚Üí Naya order submit karna POST /login Body: { "username": "user", "password": "pass" } ‚Üí Login authentication

üîí Hacking Mein:

POST requests ko aksar target kiya jata hai:

  • SQL Injection - Form fields mein
  • XSS Attacks - User input mein
  • Mass Assignment - Extra parameters bhej kar
  • Authentication Bypass - Credentials manipulation
  • CSRF Attacks - Cross-site request forgery
  • File Upload - Malicious files upload karna
PUT
‚úó Not Safe ‚úì Idempotent

üéØ Maqsad (Purpose):

Kisi khaas resource ko update karna ya agar wo mojood na ho to use banana.

‚≠ê Khaasiyat (Characteristics):

  • Idempotent: Kai baar same request se wahi asar
  • Poore resource ko replace kar deta hai
  • Agar resource exist nahi karta, to naya ban jata hai
  • Complete data bhejni hoti hai
  • Server ki state change hoti hai

üí° Misaal (Examples):

PUT /users/123 Body: { "name": "Ali Khan", "email": "alikhan@test.com", "age": 25 } ‚Üí User 123 ke poore profile ko update PUT /products/456 Body: { "name": "Phone", "price": 50000 } ‚Üí Product 456 ko replace karna

üîí Hacking Mein:

PUT vulnerabilities:

  • Broken Access Control - Unauthorized updates
  • Privilege Escalation - Admin rights set karna
  • Data Tampering - Sensitive fields modify karna
  • Mass Assignment - Extra fields bhej kar
  • File Overwrite - Malicious files upload
DELETE
‚úó Not Safe ‚úì Idempotent

üéØ Maqsad (Purpose):

Kisi khaas resource ko delete karna.

‚≠ê Khaasiyat (Characteristics):

  • Idempotent: Kai baar delete karne se wahi state
  • Resource permanently remove ho jata hai
  • Pehli successful request ke baad, subsequent requests koi change nahi kartin
  • Generally no request body
  • Dangerous operation - careful handling zaroori

üí° Misaal (Examples):

DELETE /users/123 ‚Üí User 123 ko delete karna DELETE /posts/789 ‚Üí Post 789 ko database se hatana DELETE /cart/items/5 ‚Üí Cart se item 5 remove karna

üîí Hacking Mein:

DELETE method vulnerabilities:

  • Broken Access Control - Doosre users ka data delete
  • Authorization Bypass - Bina permission delete
  • Data Loss - Critical data permanently remove
  • Denial of Service - Important resources delete kar ke
  • IDOR - ID change karke unauthorized deletion

‚≠ê Khaasiyat (Characteristics):

  • Partial Update: Sirf specific fields update hote hain
  • Aam taur par not idempotent (resource state par depend)
  • PUT se mukhtalif - poora resource replace nahi karta

üí° Misaal (Examples):

PATCH vulnerabilities (PUT ki tarah):

  • Privilege Escalation - Role/permission fields change

  • Business Logic Bypass - Validation skip karna
  • Partial Data Tampering - Critical fields ka manipulation
HEAD
‚úì Safe ‚úì Idempotent

üéØ Maqsad (Purpose):

GET ki tarah, lekin response body ke bagair. Sirf headers milte hain.

‚≠ê Khaasiyat (Characteristics):

  • Safe aur Idempotent - GET ki tarah
  • Sirf metadata hasil karta hai (headers)
  • Response body nahi milta
  • Bandwidth save hoti hai
  • Resource existence check karne ke liye useful

üí° Misaal (Examples):

HEAD /files/document.pdf Response Headers: Content-Type: application/pdf Content-Length: 2048000 Last-Modified: 2024-01-15 ‚Üí File ki tafseelat, file download kiye bagair HEAD /users/123 ‚Üí Check karna ke user exist karta hai ya nahi

üîí Hacking Mein:

HEAD method ka reconnaissance mein istemal:

  • Server Information - Headers se server type pata karna
  • Technology Stack - Framework aur versions detect karna
  • Resource Enumeration - Kam bandwidth mein testing
  • Cache Behavior - Caching headers analysis
  • Hidden Endpoints - Existence confirmation
OPTIONS
‚úì Safe ‚úì Idempotent

üéØ Maqsad (Purpose):

Target resource ke liye communication options bayan karna. Allowed HTTP methods ka pata lagana.

‚≠ê Khaasiyat (Characteristics):

  • Server ki supported methods batata hai
  • CORS (Cross-Origin Resource Sharing) mein istemal
  • Preflight requests ke liye
  • API capabilities discover karna
  • Safe aur idempotent operation

üí° Misaal (Examples):

OPTIONS /users Response Headers: Allow: GET, POST, HEAD, OPTIONS Access-Control-Allow-Methods: GET, POST, PUT ‚Üí Endpoint par allowed methods OPTIONS /api/v1/products ‚Üí Products endpoint ki capabilities

üîí Hacking Mein:

OPTIONS method se sensitive information:

  • Allowed Methods - Kaunse operations possible hain
  • CORS Configuration - Cross-origin policies
  • API Capabilities - Supported operations discover
  • Server Configuration - Misconfiguration detection
  • Hidden Methods - Undocumented endpoints ka pata

üìù Mukhtasar

Key Takeaways:

  • ‚Üí Method + Endpoint = Complete API request
