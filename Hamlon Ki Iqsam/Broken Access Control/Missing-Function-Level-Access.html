üéØ Missing Function Level Access Kya Hai?

"Missing Function Level Access" ka matlab hai "functions tak pohanchnay mein nuqs". Yeh "Broken Access Control" category mein aata hai, jo API hacking roadmap ka aik hissa hai.

Seedhay alfaaz mein: Iska matlab yeh hai ke aik API ya application sahih tareeqay se yeh check nahi karta ke kisi user ke paas kisi khaas function ya resource tak pohanchnay ke liye zaroori permissions hain ya nahi.

üé≠ Simple Misaal Se Samjhiye

Imagine karein aik company office building jahan mukhtalif departments hain aur har department ka apna secure area hai:

  • HR Department: Sirf HR staff enter kar sakta hai - employee records, salaries
  • Finance Department: Sirf finance team - company bank accounts, budgets
  • IT Department: Sirf IT admins - server room, system configurations

Proper Security: Har door par security guard hai jo aapki ID card check karta hai aur verify karta hai ke aap us department mein jaane ke authorized hain ya nahi.

Missing Function Level Access: Doors par locks hain lekin koi guard nahi! Agar aapko door ka code pata chal jaye ya key mil jaye, to aap kisi bhi department mein ja sakte hain - chahe aapko authorization ho ya na ho. System check hi nahi kar raha ke aap authorized hain ya nahi!

Web applications mein yeh tab hota hai jab mukhtalif users ke mukhtalif roles (maslan, administrator, editor, viewer) hain. Agar application kisi user ki role ko sahih tareeqay se verify nahi karta usay koi action perform karnay se pehlay, toh kam privileges wala user un functions tak pohanch sakta hai jo sirf ziyada privileges walay users ke liye hain.

‚ùìYeh Kaise Hota Hai?

Aksar aisa tab hota hai jab developers access control checks sirf user interface (UI) level par lagate hain, lekin API endpoint level par nahi.

Aik attacker UI ki restrictions ko bypass kar sakta hai aur seedhay API endpoints par requests bhej sakta hai jo sensitive actions perform kartay hain.

üö® Security Through Obscurity - Galat Approach!

Bohat se developers sochte hain: "Agar hum admin buttons ko UI mein hide kar dein, to users unhe access nahi kar sakte." Yeh bohot bari galti hai!

  • UI hiding sirf ek illusion hai - real security nahi
  • Browser DevTools se hidden elements dekhe ja sakte hain
  • API endpoints direct call kiye ja sakte hain
  • Burp Suite, Postman jaise tools se requests manually bheje ja sakte hain

Attack Ka Process:

  • Discovery: Attacker API endpoints discover karta hai (documentation, network traffic, URL patterns)
  • Testing: Different endpoints ko test karta hai with regular user credentials
  • Exploitation: Admin/privileged endpoints ko access kar leta hai without proper authorization
  • Abuse: Sensitive functions ko unauthorized execute karta hai

üìùDetailed Misaal - E-commerce Website

Farz karein aik e-commerce website par aik API endpoint hai:

Endpoint: /api/admin/deleteProduct

Yeh endpoint sirf administrators ke liye accessible hona chahiye taake wo products ko delete kar saken.

‚úÖ Sahi Implementation (Secure):

POST /api/admin/deleteProduct Authorization: Bearer user_token { "product_id": "123" }

Backend Process:

1. Token se user identity verify karo (Authentication) 2. User ki role check karo (Authorization) 3. Agar role === "admin", to proceed 4. Agar role !== "admin", to 403 Forbidden return karo 5. Product delete karo (if authorized)

‚ùå Missing Function Level Access Vulnerability:

POST /api/admin/deleteProduct Authorization: Bearer regular_user_token { "product_id": "123" }

Vulnerable Backend:

1. Token se user identity verify karo (Authentication) ‚úÖ 2. User ki role check karo? ‚ùå MISSING! 3. Directly product delete kar do!

Regular user successfully product delete kar sakta hai!

UI vs API - Security Gap

Yeh vulnerability is liye hota hai kyunki:

üí° Frontend (UI Level):

// React/Vue/Angular code if (user.role === 'admin') { <button onClick={deleteProduct}>Delete Product</button> }

UI mein "Delete Product" button sirf admin ko dikhta hai - regular users ko nahi.

üí° Backend (API Level) - VULNERABLE:

// Express.js example app.post('/api/admin/deleteProduct', (req, res) => { // ‚ùå No role check here! const productId = req.body.product_id; db.deleteProduct(productId); res.json({success: true}); });

Backend par koi role verification nahi! Koi bhi authenticated user is endpoint ko call kar sakta hai!

Attacker Kaise Exploit Karta Hai

üí° Attack Steps:

Step 1: Regular user login karta hai

Username: hacker@example.com Role: regular_user

Step 2: Browser DevTools open karta hai

Network tab mein admin users ki activities monitor karta hai Ya documentation se admin endpoints discover karta hai

Step 3: Burp Suite ya Postman use karta hai

POST /api/admin/deleteProduct Authorization: Bearer <regular_user_token> { "product_id": "999" }

Step 4: Success! Product deleted!

Response: { "success": true, "message": "Product deleted" }

Attacker ne successfully admin function execute kar diya, halaanke wo sirf regular user hai!

üõ°Ô∏èBachao - Defenses Aur Mitigations

"Missing Function Level Access" se bachnay ke liye, developers ko har API endpoint par mazboot access control checks laganay chahiye:

1. Authentication Aur Authorization Checks Lagana

Yaqeeni banayein ke har API request authenticated ho aur phir user ke role aur permissions ki bunyad par authorized ho.

üí° Proper Implementation:

// Express.js with middleware const requireAuth = (req, res, next) => { if (!req.session.userId) { return res.status(401).json({error: 'Not authenticated'}); } next(); }; const requireAdmin = (req, res, next) => { if (req.user.role !== 'admin') { return res.status(403).json({error: 'Forbidden'}); } next(); }; // Secure endpoint app.post('/api/admin/deleteProduct', requireAuth, requireAdmin, (req, res) => { // Now safe to proceed const productId = req.body.product_id; db.deleteProduct(productId); res.json({success: true}); });
  • Har endpoint par authentication verify karen
  • Role-based checks implement karen
  • Permission-based access control use karen
  • Middleware patterns follow karen consistency ke liye
2. Least Privilege Principle

Users ko sirf un functions aur data tak rasai milni chahiye jo unki role ke liye bilkul zaroori hain.

  • Default deny policy - explicitly allow karna zaroori
  • Minimum necessary permissions hi assign karen
  • Temporary elevated access for specific tasks
  • Regular permission audits conduct karen
  • Unused privileges revoke karen
  • Time-bound permissions implement karen
3. Role-Based Access Control (RBAC)

Wazeh roles define karein aur har role ko khaas permissions assign karein, phir inhein API level par enforce karein.

üí° RBAC Implementation:

const permissions = { guest: ['read:public'], user: ['read:public', 'read:own', 'write:own'], moderator: ['read:all', 'write:own', 'moderate:content'], admin: ['read:all', 'write:all', 'delete:all', 'manage:users'] }; function hasPermission(userRole, requiredPermission) { return permissions[userRole].includes(requiredPermission); }
  • Clear role hierarchy define karen
  • Granular permissions assign karen
  • Permission matrix maintain karen
  • Role inheritance properly manage karen
  • Dynamic role assignment with audit logging
4. API Gateway

Aik API Gateway access control policies ko markazi bananay aur enforce karnay mein madad kar sakta hai.

  • Centralized authentication aur authorization
  • Rate limiting aur throttling
  • Request validation aur sanitization
  • Logging aur monitoring centralization
  • Policy enforcement consistency
  • API versioning aur deprecation management
5. Defense in Depth

Multiple layers of security implement karen:

  • Layer 1: Network level - IP filtering, firewalls
  • Layer 2: API Gateway - rate limiting, validation
  • Layer 3: Application level - authentication
  • Layer 4: Endpoint level - authorization checks
  • Layer 5: Data level - field-level permissions
  • Layer 6: Monitoring - anomaly detection
6. Secure API Design Patterns

API endpoints ko secure design principles ke sath banayein:

üí° Secure URL Patterns:

// ‚ùå BAD - Role in URL (predictable) /api/admin/deleteProduct /api/user/viewProfile // ‚úÖ GOOD - Action-based with authorization /api/products/delete (with role check in backend) /api/profile (user determined from session)
  • RESTful best practices follow karen
  • Predictable patterns avoid karen
  • Versioning for security updates
  • Comprehensive documentation with security notes
7. Comprehensive Logging Aur Monitoring

Har authorization attempt ko log karen:

  • Successful aur failed authorization attempts log karen
  • Unusual access patterns detect karen
  • Privilege escalation attempts alert karen
  • Real-time monitoring dashboards
  • Automated alerting systems
  • SIEM integration
  • Regular audit log reviews

Continuous security assessment conduct karen:

Modern frameworks ki built-in security features leverage karen:

‚ö†Ô∏è Common Mistakes To Avoid

  • UI-only security: Sirf frontend par buttons hide karna - backend vulnerable hai
  • Security through obscurity: Endpoints ko secret rakhna - always discovered
  • Inconsistent checks: Some endpoints secure, others not
  • Hard-coded roles: Code mein roles hard-code karna instead of database
  • Client-side validation only: Server par koi checks nahi
  • Token without validation: Token accept karna without proper verification
  • Missing edge cases: Special scenarios mein authorization skip ho jaata hai

üí° Golden Rule

"Never trust the client. Always verify authorization on the server for EVERY sensitive operation."

UI restrictions sirf user experience ke liye hain - security ke liye nahi. Real security hamesha server-side implement karni chahiye, har request ke liye, har endpoint par.

Authentication (user kaun hai) + Authorization (user ko kya karne ki permission hai) = Complete Security

üìå Khulasa (Conclusion)

Missing Function Level Access ek critical vulnerability hai jo tab hoti hai jab applications proper authorization checks implement nahi karti hain har function/endpoint par.

Yeh vulnerability aksar is liye hoti hai kyunki developers sirf UI level par access control lagaate hain aur backend API endpoints ko improperly secure karte hain. Attackers UI restrictions ko asaani se bypass kar ke sensitive functions ko directly access kar sakte hain.

Prevention strategies: Har endpoint par comprehensive authentication + authorization checks, Role-Based Access Control (RBAC), Least Privilege Principle, API Gateway for centralized security, aur multi-layer defense approach implement karen.

Yaad rakhein: Security through obscurity kaam nahi karta. Real security server-side enforcement se aati hai. Har sensitive function, har API endpoint ko properly secure karen with mandatory authorization checks!
