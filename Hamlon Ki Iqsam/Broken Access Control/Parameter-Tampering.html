üéØ Parameter Tampering Kya Hai?

"Parameter Tampering" API Hacking roadmap ke "Hamlon Ki Iqsam" (Types of Attacks) section mein "Broken Access Control" ke under aata hai.

Parameter Tampering ek qisam ka hamla hai jahan ek hacker client (jaise web browser ya mobile app) aur server ke darmiyan exchange hone wale parameters ko manipulate karta hai. Iska maqsad application logic ko modify karna, security checks ko bypass karna, ya ghair-ikhtiyari access haasil karna hota hai.

üé≠ Simple Misaal Se Samjhiye

Imagine karein aap ek restaurant mein hain aur aap waiter ko apna order likhit form par dete hain:

Normal Order Form:

  • Item: Burger
  • Price: $10
  • Quantity: 1
  • Total: $10

Parameter Tampering: Agar restaurant (server) form ko check kiye baghair bill calculate kare, to aap (attacker) form par price ko $10 se $1 kar sakte hain. Restaurant sirf form par jo likha hai us par trust karke bill bana de!

Web applications mein yeh tab hota hai jab server client se aane wale parameters (price, user_id, role, etc.) par blind trust karta hai baghair verify kiye!

üìçParameters Kahan Mil Sakte Hain?

Yeh parameters mukhtalif jaghon par mil sakte hain:

1. URL Parameters (Query Strings)

URL mein ? ke baad wale hisse:

üí° Examples:

www.example.com/item?id=123&quantity=1 www.shop.com/checkout?price=100&discount=0 www.bank.com/transfer?from=123&to=456&amount=1000 www.social.com/profile?user_id=789&role=user
  • Easily visible in browser address bar
  • Browser history mein save hote hain
  • Attacker easily modify kar sakta hai
  • GET requests mein common

2. Form Fields (POST Data)

Hidden ya dikhne wale fields jo POST requests ke through submit kiye jaate hain:

üí° HTML Form Example:

<form method="POST" action="/checkout"> <input type="hidden" name="price" value="100"> <input type="hidden" name="user_id" value="123"> <input type="text" name="quantity" value="1"> <button type="submit">Buy</button> </form>

Attacker browser DevTools se hidden fields ko modify kar sakta hai!

  • Hidden fields ko "hidden" samajh kar trust karna galti hai
  • Browser inspect tool se easily visible
  • Burp Suite se intercept aur modify kiye ja sakte hain

3. HTTP Headers

Request headers mein sensitive information:

üí° Common Headers:

Cookie: session_id=abc123; user_role=admin Authorization: Bearer eyJhbGci... X-User-ID: 123 X-Auth-Token: secret_token_xyz X-Role: administrator
  • Cookie headers easily tampered
  • Custom headers vulnerable to manipulation
  • Authorization headers ko modify kiya ja sakta hai
  • User agents aur referrers spoofed ho sakte hain

4. JSON/XML Body

Request body mein bheja gaya data, jo khaas tor par API calls mein common hai:

üí° JSON Body Example:

POST /api/checkout Content-Type: application/json { "user_id": 123, "item_id": "ABC", "price": 100, "quantity": 1, "is_admin": false, "discount_applied": false }
  • REST APIs mein bohot common
  • Postman/Burp Suite se easily modified
  • Complex nested objects vulnerable
  • XML injection bhi possible

5. Cookies

Browser cookies mein stored session data:

üí° Cookie Tampering:

// Original cookie: user_id=123; role=user; premium=false // Tampered cookie: user_id=1; role=admin; premium=true
  • Browser DevTools se easily editable
  • Cookie editors available
  • Base64 encoded cookies decode kiye ja sakte hain
  • Weak encryption breakable hai

‚öôÔ∏èYeh Kaise Kaam Karta Hai?

Iska bunyadi khayal yeh hai:

üí° Attack Process:

Step 1: Request Intercept Karna

Attacker apne client se server ko bheji gayi request ko intercept karta hai Tools: Burp Suite, OWASP ZAP, Browser DevTools

Step 2: Parameters Modify Karna

Request mein ek ya zyada parameters ko badalta hai Price, user_id, role, permissions, etc.

Step 3: Modified Request Bhejana

Modified request ko server ko bhej deta hai

Step 4: Server Response

Agar server in parameters ko theek se validate ya re-verify nahi karta, toh woh request ko badle hue values ke saath process kar sakta hai!

üö® Core Problem

Agar server in parameters ko theek se validate ya re-verify nahi karta, toh woh request ko badle hue values ke saath process kar sakta hai, jis se ghair-mutawaqqe nateeje nikal sakte hain.

  • Client-side validation easily bypassable
  • Server ko har parameter verify karna chahiye
  • Critical data client se nahi aana chahiye

üìùParameter Tampering Ki Misaalein

1. Price Manipulation (Qeemat Mein Hera Pheri)

Scenario: Online shopping cart mein price tampering

üí° Normal Request:

POST /checkout Content-Type: application/json { "item_id": "ABC", "price": 100, "quantity": 1 }

Total: $100

üí° Tampered Request:

POST /checkout Content-Type: application/json { "item_id": "ABC", "price": 1, // ‚ö†Ô∏è Changed from 100 to 1! "quantity": 1 }

Agar server client ki price par trust kare, hacker $1 mein product khareed lega!

Kya Hota Hai: Ek hacker is request ko rok kar price parameter ko 1 mein tabdeel kar sakta hai. Agar server client ki taraf se di gayi qeemat par aankh band kar ke bharosa karta hai aur apni database se dobara uski tasdeeq nahi karta, toh hacker us cheez ko $1 mein khareed sakta hai $100 ke bajaye.

  • Impact: Direct financial loss
  • Fix: Server ko database se actual price fetch karni chahiye

2. Access Control Bypass (Access Control Ko Bypass Karna)

Scenario: User ID manipulation se doosre user ka data access

üí° Normal Request:

GET /user?id=5 Authorization: Bearer user_token

User 5 ki apni profile dekh raha hai ‚úÖ

üí° Tampered Request:

GET /user?id=1 // ‚ö†Ô∏è Changed from 5 to 1 (admin ID) Authorization: Bearer user_token

Agar authorization check nahi hai, to User 1 (admin) ka data accessible!

Kya Hota Hai: Ek hacker id=5 ko id=ADMIN ya id=1 (ye farz karte hue ke 1 kisi administrator ki ID ho sakti hai) mein tabdeel kar sakta hai. Agar application yeh theek se check nahi karti ke request karne wale user ke paas dusre users ka data (khaas tor par admin ka) dekhne ka ikhtiyar hai ya nahi, toh hacker ghair-ikhtiyari tor par dusre accounts ki maloomat ya hatta ke administrative functions tak rasai haasil kar sakta hai.

  • Relation: Yeh IDOR (Insecure Direct Object References) se qareeb se mutaliq hai
  • Impact: Unauthorized data access, privacy breach
  • Fix: Har request par ownership verify karen

3. Privilege Escalation (Ikhtiyarat Badhana)

Scenario: User role manipulation

üí° Normal Profile Update:

PUT /api/profile { "name": "John Doe", "email": "john@example.com", "is_admin": false }

üí° Tampered Request:

PUT /api/profile { "name": "John Doe", "email": "john@example.com", "is_admin": true // ‚ö†Ô∏è Changed to true! }

Agar server ko field-level checks nahi hain, user apne aap ko admin bana lega!

Kya Hota Hai: Ek user profile update request mein is_admin=false jaisa parameter shamil ho sakta hai. Ek hacker request mein is_admin=false ko is_admin=true mein tabdeel kar sakta hai. Agar server ke paas user roles ko badalne ke liye mazboot server-side checks nahi hain, toh hacker apne ikhtiyarat ko administrator tak badha sakta hai.

  • Impact: Complete system compromise
  • Fix: Critical fields ko user input se accept na karen
  • Fix: Role changes sirf authorized admins kar saken

4. Transaction Manipulation (Lenden Mein Hera Pheri)

Scenario: Banking transfer mein account number tampering

üí° Normal Transfer:

POST /api/transfer { "from_account": "123456", "to_account": "789012", // Legitimate recipient "amount": 1000 }

üí° Tampered Transfer:

POST /api/transfer { "from_account": "999999", // ‚ö†Ô∏è Changed to victim's account! "to_account": "666666", // ‚ö†Ô∏è Attacker's account "amount": 50000 // ‚ö†Ô∏è Increased amount! }

Agar proper authorization nahi hai, attacker kisi aur ke account se apne account mein transfer kar dega!

Kya Hota Hai: Banking application mein, transfer request mein bhejnewale aur wasool karne wale account numbers specify kiye ja sakte hain. Ek hacker mumkin hai ke wasool karne wale account number ko apne control wale account number mein tabdeel kar de.

  • Impact: Financial fraud, money theft
  • Fix: Strong authentication aur authorization
  • Fix: Transaction verification (OTP, 2FA)
  • Fix: Logged-in user ke account se hi transfer allow karen

5. Discount/Coupon Manipulation

Scenario: Discount percentage ko manipulate karna

üí° Tampered Discount:

POST /api/apply-discount { "cart_id": "ABC123", "coupon_code": "SAVE10", "discount_percent": 99 // ‚ö†Ô∏è Should be 10, changed to 99! }
  • Coupon validity bypass
  • Discount percentage manipulation
  • Multiple coupon application
  • Expired coupons reactivation

6. Quantity/Inventory Manipulation

Scenario: Limited stock items ko bypass karna

üí° Tampered Quantity:

POST /api/order { "product_id": "LIMITED_001", "quantity": 1000, // ‚ö†Ô∏è Only 10 in stock! "stock_check": "passed" // ‚ö†Ô∏è Fake parameter! }
  • Out-of-stock items order karna
  • Maximum order limit bypass
  • Inventory validation skip karna

üõ°Ô∏èParameter Tampering Se Kaise Bacha Jaye?

1. Server-Side Validation (Server Ki Taraf Se Tasdeeq)

Yeh sabse ahem hifazat hai. Client se aane wale data par kabhi bharosa na karein. Tamam sensitive parameters ko server-side par validate, re-verify ya regenerate kiya jana chahiye.

üí° Secure Implementation:

// ‚ùå BAD - Trusting client price app.post('/checkout', (req, res) => { const price = req.body.price; // Dangerous! processPayment(price); }); // ‚úÖ GOOD - Fetching from database app.post('/checkout', (req, res) => { const itemId = req.body.item_id; const price = db.getPrice(itemId); // Server determines price processPayment(price); });
  • Never trust client input for critical values
  • Prices, permissions, roles server se fetch karen
  • User IDs session se len, parameter se nahi
  • Critical calculations server-side karen
2. Strong Access Control Checks

Har sensitive operation par mazboot authorization checks laagu karein taake yeh yaqeeni ho sake ke authenticated user ke paas specified resource par action perform karne ke liye zaruri permissions hain.

üí° Authorization Check:

app.get('/user/:id', requireAuth, (req, res) => { const requestedId = req.params.id; const currentUserId = req.session.userId; // Ownership check if (requestedId !== currentUserId && !req.user.isAdmin) { return res.status(403).json({error: 'Forbidden'}); } const user = db.getUser(requestedId); res.json(user); });
  • Har request par ownership verify karen
  • Role-based access control implement karen
  • Resource-level permissions check karen
  • Default deny approach use karen
3. Input Sanitization (Input Ki Safai)

Tamam user input ko saaf karein taake injection attacks se bacha ja sake aur data mutawaqqe formats ke mutabiq ho.

  • Type validation: Data types verify karen
  • Range validation: Min/max values check karen
  • Format validation: Expected patterns match karen
  • Whitelist approach: Sirf allowed values accept karen
  • SQL injection prevention: Parameterized queries
  • XSS prevention: HTML encoding
4. Session Management (Session Ka Intezaam)

Session token manipulation ko rokne ke liye mazboot session management techniques istemal karein.

  • Cryptographically secure session IDs
  • Session data server-side store karen
  • HttpOnly aur Secure flags use karen cookies mein
  • Session hijacking se bachen
  • Regular session rotation
  • Session timeout implement karen
5. Encrypt Sensitive Parameters

Nihayat sensitive data ke liye, parameters ko encrypt karna hifazat ki ek mazeed teh laga sakta hai, halanke server-side validation ab bhi sabse ahem hai.

üí° Encryption Example:

// Instead of plain IDs: /user?id=123 ‚ùå // Use encrypted tokens: /user?token=a7f8d9e2b3c4... ‚úÖ // Server decrypts and validates: const userId = decrypt(token); if (!isValidUser(userId)) return 403;
  • Sensitive IDs encrypt karen
  • JWTs use karen with proper signing
  • Tokens time-bound banayein
  • One-time tokens use karen critical operations ke liye
6. Use Anti-CSRF Tokens

Halanke yeh bunyadi tor par Cross-Site Request Forgery ke liye hain, anti-CSRF tokens attackers ke liye malicious requests banana mushkil bana sakte hain.

  • Har form mein unique CSRF token
  • Token server-side validate karen
  • SameSite cookie attribute use karen
  • Double submit cookie pattern
7. Rate Limiting Aur Throttling

Automated parameter tampering attacks ko slow karen:

  • Request rate limits per user/IP
  • Brute force attempts detect karen
  • Temporary bans for suspicious activity
  • CAPTCHA after failed attempts
  • API quotas enforce karen
8. Comprehensive Logging Aur Monitoring

Parameter tampering attempts ko detect karen:

  • Har request ko log karen with parameters
  • Unusual parameter values detect karen
  • Failed validation attempts track karen
  • Real-time alerting for suspicious patterns
  • Audit trails maintain karen
  • SIEM integration for analysis
9. Web Application Firewall (WAF)

WAF parameter tampering patterns detect kar sakta hai:

  • Malicious parameter patterns block karen
  • SQL injection attempts prevent karen
  • XSS attacks filter karen
  • Custom rules define karen
  • Virtual patching for known vulnerabilities
10. Regular Security Testing

Parameter tampering vulnerabilities dhoondhein:

  • Penetration testing specifically for parameter tampering
  • Automated security scanning tools
  • Code reviews focusing on input validation
  • Bug bounty programs
  • Fuzzing tests for unexpected inputs

‚ö†Ô∏è Common Developer Mistakes

  • Client-side validation only: JavaScript validation easily bypass
  • Hidden fields ko secure samajhna: Easily visible in HTML
  • Price/amount client se lena: Always fetch from database
  • Role/permissions client se accept karna: Server se determine karen
  • Session data cookies mein plain text: Encrypt ya server-side store karen
  • URL parameters par bharosa: Easily modified

üí° Golden Rules

Rule #1: "Never trust the client." Client se aane wala har parameter potentially malicious ho sakta hai.

Rule #2: "Validate everything server-side." Client-side validation sirf user experience ke liye hai - security ke liye nahi.

Rule #3: "Critical data server se fetch karen." Prices, permissions, roles kabhi client se na len.

Rule #4: "Default deny approach." Explicitly allow karna zaroori, implicit trust nahi.

üìå Khulasa (Conclusion)

Parameter Tampering ek critical vulnerability hai jo tab hoti hai jab applications client-side input par blind trust karti hain. Attackers easily request parameters ko modify kar ke prices manipulate kar sakte hain, unauthorized access haasil kar sakte hain, privileges escalate kar sakte hain, aur business logic ko bypass kar sakte hain.

Yeh vulnerability is bunyadi usool ko ujagar karta hai ke "client na-qabil-e-bharosa hai". Tamam ahem faisle aur validations server-side par hone chahiye taake attackers ko application ke rawaiye ko manipulate karne se roka ja sake.

Prevention strategy: Comprehensive server-side validation, strong authorization checks, secure session management, input sanitization, encryption, aur continuous monitoring implement karen. Yaad rakhein: client-side security sirf illusion hai - real security hamesha server par enforce hoti hai!
