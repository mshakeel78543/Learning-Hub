üéØ Broken Authentication Logic Kya Hai?

"Broken Authentication Logic" ka topic, jo API Hacking roadmap ke "Shurwati Hamle" (Initial Attacks) section mein "Authentication Bypass" ke tehat aata hai, iska matlab hai ke user ki pehchan verify karne ke tareeqe mein kharabiyan hain.

Seedhe lafzon mein, iska matlab hai ke authentication mechanism, jo aapki pehchan ki tasdeeq ke liye banaya gaya hai, usmein tarkeebi kamzoriyan ya design ki ghaltiyan hain jin ka faida utha kar ek hacker usse bypass kar sakta hai.

üîç Ahem Farq

Hacker credentials ko seedhe crack karne ke bajaye, authentication process ko dhokha dene ya usse ghoom kar nikalne ki koshish karta hai. Yeh masla passwords ke kamzor hone ka nahi balki authentication flow ke code ya design mein ghaltiyan ka hai.

üêõLogic Flaws (Tarkeebi Ghalatiyan) Ki Types

Asal masla zaroori nahi ke kamzor passwords hon, balki authentication flow ke code ya design mein ghaltiyan hoti hain:

1. Improper State Management (Ghalat State Intezaam)

Application shayad user ki authentication state ko sahih tareeqe se track na kare, jis se ek hacker purane session tokens ko dobara istemal kar sake ya steps ko bypass kar sake.

üí° Vulnerable Scenario:

User logs in ‚Üí Session created with state="pending" User verifies email ‚Üí State should change to "verified" But application doesn't check state before granting access

Result: User with "pending" state can access protected resources

  • Session state transitions properly handle nahi hote
  • Intermediate states ka validation missing
  • State rollback possible hai
  • Concurrent requests se state corruption

2. Flawed Multi-Factor Authentication (MFA) Implementation

Agar MFA maujood hai, lekin ek hacker phir bhi doosre factor ke bagair kisi aur endpoint se ya request mein hera pheri karke log in kar sakta hai.

üí° MFA Bypass Examples:

POST /login ‚Üí Username/Password accepted ‚Üí MFA required But attacker directly accesses: GET /api/dashboard ‚Üí Access granted! (MFA skipped)

Ya

Cookie mein mfa_verified=false ‚Üí Manually change to true
  • MFA sirf kuch endpoints par enforced
  • MFA status client-side par maintained
  • Backup authentication methods insecure
  • "Remember this device" feature ka misuse

3. Weak Password Reset Mechanisms

Agar password reset process ko aasani se manipulate kiya ja sakta hai (maslan, request mein user ID badal kar ya reset tokens ko sahih tareeqe se validate na karke).

üí° Reset Vulnerabilities:

POST /reset-password { "email": "attacker@example.com", "user_id": 123, ‚Üê Change to victim's ID "new_password": "hacked123" }

Server user_id par rely karta hai, email verify nahi karta

  • Reset tokens predictable hain
  • Token expiry properly check nahi hoti
  • Same token multiple times use ho sakta hai
  • User verification insufficient hai
  • Reset links sensitive data expose karte hain

4. Bypassing Steps In A Workflow

Ek hacker zaroori authentication steps ko chhod kar, jaise email verification, seedhe authenticated functions tak pahunch sakta hai.

üí° Step Bypass Example:

Expected Flow: 1. Register ‚Üí 2. Verify Email ‚Üí 3. Complete Profile ‚Üí 4. Access Dashboard

Bypass:

1. Register ‚Üí 2. Skip verification ‚Üí 4. Direct /dashboard access
  • Sequential workflow enforcement missing
  • Step completion validation na hona
  • Direct endpoint access possible
  • State transition checks absent

5. Insecure Account Creation

Agar account banane ki logic ka galat istemal karke administrator ke taur par register kiya ja sake ya maujooda user accounts ko overwrite kiya ja sake.

üí° Registration Exploits:

POST /register { "username": "newuser", "password": "pass123", "role": "admin" ‚Üê Attacker sets admin role }

Agar backend role ko validate nahi karta

  • Role assignment client se control hota hai
  • Existing username overwrite ho sakta hai
  • SQL injection via registration form
  • Mass assignment vulnerabilities
  • No email/phone verification

6. Session Fixation

Attacker pehle se ek session ID set kar leta hai, phir victim ko us session use karne par majboor karta hai.

üí° Session Fixation Attack:

1. Attacker gets session ID: ABC123 2. Sends victim: http://site.com/?session=ABC123 3. Victim logs in with this session 4. Attacker uses ABC123 to access victim's account
  • Login par naya session generate nahi hota
  • Session ID URL parameters se accept hoti hai
  • Old sessions invalidate nahi hote

7. JWT Token Vulnerabilities

JSON Web Tokens ki improper implementation se authentication bypass possible hai.

üí° JWT Flaws:

// None algorithm attack {"alg":"none","typ":"JWT"}

Ya

// Change user_id in payload {"user_id":1,"role":"admin"}

Agar signature verify nahi hota

  • "none" algorithm accept karna
  • Weak signing keys
  • No token expiration
  • Signature verification skip hona
  • Key confusion attacks

8. OAuth/SSO Implementation Flaws

Third-party authentication mechanisms mein logic errors.

üí° OAuth Vulnerabilities:

// Redirect URI manipulation redirect_uri=https://attacker.com

Ya

// State parameter missing (CSRF)
  • Open redirect vulnerabilities
  • Missing state parameter validation
  • Token leakage via referer headers
  • Improper scope validation

üîåAPIs Mein Iske Zahir Hone Ki Misalein

1. Manipulating API Endpoints

Ek API mein authenticated users ke liye ek endpoint ho sakta hai aur unauthenticated users ke liye ek alag endpoint. Agar ek hacker system ko yeh samjhne par majboor kar sakta hai ke woh authenticated hai, to yeh broken authentication logic hai.

üí° Endpoint Manipulation:

Authenticated: GET /api/v1/user/profile Public: GET /api/v1/public/data

Attack:

GET /api/v1/user/profile (without valid auth) Server doesn't properly check authentication ‚Üí Sensitive data exposed!

2. Predictable Session Tokens

Agar session tokens qabil-e-peshgoi tareeqe se banaye jaate hain, to ek hacker valid tokens ka andaza laga sakta hai aur login ko bypass kar sakta hai.

üí° Predictable Token Patterns:

User 1: session_token=1000 User 2: session_token=1001 User 3: session_token=1002

Attacker tries: session_token=1003, 1004, 1005...

Ya

token = MD5(username + timestamp)

Agar algorithm known hai to prediction possible

3. Lack Of Proper Authorization After Authentication

Kabhi kabhi, authentication kaam kar sakti hai, lekin application yeh theek se check karne mein nakam rehti hai ke authenticated user ko ek khaas action karne ki ijazat hai ya nahi.

üí° Missing Authorization:

User logged in successfully (Authentication ‚úì) GET /api/admin/delete-user/123 Server checks: User is authenticated? Yes Server doesn't check: Is user an admin? No check! ‚Üí Regular user can delete users

4. Parameter Tampering

Request parameters ko modify kar ke authentication bypass karna.

üí° Parameter Attack:

POST /api/transfer-money { "from_account": "123", "to_account": "456", "amount": 1000, "authenticated": true ‚Üê Attacker adds this }

5. Race Conditions In Authentication

Concurrent requests se authentication checks bypass karna.

üí° Race Condition:

Thread 1: POST /login (credentials sent) Thread 2: GET /protected-resource

Agar Thread 2 jaldi execute ho jaye before full authentication validation

üö® Yeh Itni Khatarnak Kyun Hai?

  • Complete Account Takeover: Hackers user accounts, jismein administrator accounts bhi shamil hain, par mukammal qabza kar sakte hain
  • Unauthorized Access: Woh sensitive data tak rasai hasil kar sakte hain, user ki taraf se actions kar sakte hain, ya poore system par qabza kar sakte hain
  • Difficult to Detect: Yeh logic flaws chote ho sakte hain aur standard security scanners se pakde nahi ja sakte
  • Manual Review Required: Application ke unique authentication flow ko samajhne ki zaroorat hoti hai
  • Widespread Impact: Ek flaw se hazaron users affected ho sakte hain

üõ°Ô∏èKaise Bachao Ho Sakta Hai

Developers ko apni authentication aur session management mechanisms ko ehtiyat se design aur sakhti se test karna chahiye:

1. Proper State Management

Authentication state ko server-side par securely manage karen.

  • State transitions ko strictly enforce karen
  • Har request par state validate karen
  • Client-side state par trust na karen
  • Session data ko encrypt karen
  • State changes ko log karen
2. Comprehensive MFA Implementation

Multi-factor authentication ko properly implement karen.

  • MFA ko sabhi sensitive endpoints par enforce karen
  • MFA bypass ke alternate routes close karen
  • Backup codes securely generate karen
  • "Remember device" feature carefully implement karen
  • MFA status server-side track karen
3. Secure Password Reset

Password reset process ko secure banayein.

  • Cryptographically secure tokens generate karen
  • Short expiry time set karen (15-30 minutes)
  • One-time use tokens implement karen
  • User identity properly verify karen
  • Email/SMS verification zaroori karen
4. Workflow Enforcement

Sequential steps ko properly enforce karen.

  • Step completion tracking implement karen
  • Direct endpoint access prevent karen
  • State machine pattern use karen
  • Prerequisite checks add karen
  • Workflow violations log karen
5. Secure Session Management

Session tokens ko securely handle karen.

  • Cryptographically random tokens generate karen
  • Login par naye session banayein
  • Logout par sessions invalidate karen
  • Session expiry implement karen
  • Concurrent session limits set karen
6. Input Validation

Sabhi inputs ko properly validate karen.

  • Server-side validation mandatory
  • Whitelist approach use karen
  • Parameter tampering prevent karen
  • Mass assignment vulnerabilities fix karen
  • Type checking enforce karen
7. Regular Security Audits

Authentication logic ko regularly review karen.

  • Code reviews conduct karen
  • Penetration testing schedule karen
  • Automated security scans run karen
  • Logic flow diagrams banayein
  • Security team feedback len
8. Principle Of Least Privilege

Minimum required permissions grant karen.

  • Default permissions minimal rakhen
  • Role-based access control implement karen
  • Permission escalation monitor karen
  • Regular permission audits karen

üîçTesting Methodology

1. State Transition Testing

Authentication workflow ke har state ko test karen.

  1. Sabhi possible states identify karen
  2. State transitions ko map karen
  3. Invalid transitions try karen
  4. State rollback attempts karen
  5. Concurrent state changes test karen

2. Endpoint Authorization Testing

Har endpoint par proper authorization check karen.

  1. Unauthenticated access try karen
  2. Different user roles test karen
  3. Cross-user access attempts karen
  4. HTTP method tampering try karen
  5. Parameter manipulation test karen

3. Session Management Testing

Session handling ko thoroughly test karen.

  1. Session fixation test karen
  2. Session token predictability check karen
  3. Session expiry validate karen
  4. Concurrent sessions test karen
  5. Session invalidation verify karen

4. Token Analysis

Authentication tokens ko analyze karen.

  1. JWT tokens decode karen
  2. Token signatures verify karen
  3. Token expiry check karen
  4. Token tampering try karen
  5. Algorithm confusion test karen

‚ö†Ô∏è Ethical Guidelines

Authentication logic testing sirf authorized environments par karein. Production systems par unauthorized testing illegal hai aur serious legal consequences ho sakte hain. Hamesha proper authorization, bug bounty programs, ya signed agreements ke saath testing karein.

üí° Ahem Nuqta

Broken Authentication Logic modern applications mein sab se critical vulnerabilities mein se ek hai. Yeh sirf technical implementation ka masla nahi balki design aur architecture decisions ka natija hai. Developers ko chahiye ke wo authentication flow ko design karte waqt security-first approach apnayen aur har possible attack scenario ko consider karen.

üìå Khulasa

Broken Authentication Logic ek complex vulnerability hai jo implementation flaws, design mistakes, aur improper security practices ka natija hai. Yeh credentials crack karne se zyada tehzeeb-daar lekin equally dangerous attack vector hai.

Successful defense ke liye comprehensive approach chahiye: secure design, proper implementation, regular testing, aur continuous monitoring. Authentication har application ki pehli defense line hai - agar yeh toot jaye to baqi saari security measures bekaar hain.

Yaad rakhein: Security by design, not by afterthought. Authentication logic ko shuru se hi security considerations ke saath design karen, na ke baad mein patch karen.
